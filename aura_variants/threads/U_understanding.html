Models that show their work. Code intelligence that can be audited, not merely trusted.

A language model that produces a fix is a hypothesis. A symbolic system that proves the fix preserves the invariants is evidence. The lab studies how to compose the two so that developers receive a result they can check, not just an answer they have to take on faith.

§ 1 · Why this matters

A correct answer with no explanation is a liability, not an asset.

The most consequential decision a developer makes about an AI suggestion is whether to merge it. That decision rests on something the system has not, until recently, been built to provide: an account of why.

When a code model proposes a patch, the developer needs to know which fields it inspected, which invariants it implicitly assumed, and which test cases would still fail. When a summarisation model condenses 200 lines into one sentence, the reader needs to know whether the sentence emerged from the function's body or merely from the function's name.

A model that cannot explain itself cannot be debugged.

Industry-grade interpretability is also a regulatory inevitability. The 2026 NIST AI Risk Management addendum on developer-assistance tools requires traceable rationales for any model whose output enters a release pipeline. Whether or not your jurisdiction has caught up, the case for explainability is already a case for compliance.

Our research treats interpretability as a research problem, not a feature you bolt onto a finished model. We pursue two complementary tracks: mechanistic — what features the model has learned and where they live — and symbolic — pairing neural predictions with grammars, type systems, and program analysers that produce explanations a developer can step through.

Two views of the same explanation
Neural
attention(merge, [list, len, half])
attention(left, [recur, list[:mid]])
attention(right, [recur, list[mid:]])
"the model attended to these tokens"
Symbolic
recursive_descent(merge_sort)
  base : len(a) ≤ 1
  split: a[:mid] ; a[mid:]
  combine: merge(L, R)
"the program analyser proved these"
Neither track alone is enough. Attention alone is correlational. A symbolic trace alone is brittle. The hybrid is both grounded and complete. Read the ICPC ERA paper for the formal account.
§ 2 · What is hard

Three honest reasons interpretability for code is not solved.

2.1 · Faithfulness

An explanation is not a story.

It is easy to generate a plausible English rationalisation of any answer; it is hard to guarantee the rationalisation reflects the computation that produced the answer. We need explanations whose truth conditions can be checked against the model's internal state.

2.2 · Granularity

Tokens are the wrong unit.

Developers reason about features — invariants, types, dataflow paths — not subword pieces. Token-level attention maps look impressive in screenshots and explain nothing to the reviewer who has to merge the patch.

2.3 · Cost

Symbolic analysis does not scale on its own.

Whole-program analysis on 1 MLoC takes hours. Pure neural prediction takes milliseconds and is wrong in surprising ways. The compositional question is how to spend symbolic effort exactly where the neural signal is uncertain — and not elsewhere.

§ 3 · How we ensure it

Method — three explanation primitives, each falsifiable.

  • Feature-level attribution
    We probe trained models for human-meaningful features (invariants, types, dataflow), not just tokens. Output: a per-feature score the developer can inspect.
  • Neurosymbolic decoding
    The model proposes; the symbolic system constrains. Type-correct, AST-valid, and invariant-preserving outputs by construction — not by post-hoc filtering.
  • Falsifiable explanations
    Every explanation comes with a test of itself: a counterfactual perturbation that should change the prediction if the explanation is correct.
  • Cost-aware analysis
    Symbolic effort is spent only where neural uncertainty is high. Cheap explanations stay cheap; expensive ones are budget-aware.
neurosym_decode.py · excerpt 
# Neurosymbolic decoding: model proposes, analyser constrains.
from aura.interp import CodeModel, TypeChecker, Trace

model    = CodeModel.load("distill-220m")
checker  = TypeChecker("python-3.12")

def propose(prompt):
    candidates = model.beam(prompt, n=8)
    # Only consider type-correct, AST-valid outputs.
    valid = [c for c in candidates if checker.ok(c)]
    return valid[0] if valid else None

trace = Trace.record(propose, on="my_repo/buggy_func.py")
# The trace contains every model logit, every checker
# verdict, every constraint that pruned a candidate.
trace.explain()            # human-readable rationale
trace.counterfactual()     # perturbation that should flip the answer

§ 5 · For practitioners

What you can use today.

  • aura-lab/interp-toolkit
    Drop-in probe that attaches to common code models (CodeBERT, CodeT5+, CodeLlama) and produces feature-level attribution plus a falsifying counterfactual.
  • Type-aware decoding wrapper
    Sits between any HuggingFace code model and your output. Guarantees AST validity and type-correctness; logs everything it pruned.
  • "Explanation contract" template
    A YAML schema your team can adopt to specify what an AI assistant must explain before its output enters CI. Adopted by 4 industrial partners as of 2026.
§ 6 · For researchers

Open problems.

  • Faithful feature probes
    Probes whose results we can prove reflect the model's computation, not just correlate with the output. Provably-faithful probes for code remain an open problem.
  • Compositional neurosymbolic decoding
    When to call the analyser, with what budget, on what scope. A theory of when to be symbolic.
  • User study designs
    Empirical evaluations of whether feature-level explanations actually change developer decisions. The interpretability literature is full of pictures and short of A/B tests.
  • Benchmark · interpretability for code
    A community-built corpus of code-AI explanations with human gold-labels for did this explanation help. We are seeding it; please contribute.
