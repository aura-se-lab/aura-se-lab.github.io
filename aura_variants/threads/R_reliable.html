Thread R · ReliableCausality, robustness & honest evaluation

Numbers that survive scrutiny. Evaluation that measures what matters, not what's easy.

An LLM judge agrees with itself across runs. An LLM judge agrees with another LLM judge. Neither fact answers the question that matters: does it agree with a competent human reviewer of code? Our work on TSE 2026 makes that question precise — and finds the disagreement is much larger than the literature suggests.

The reason the field reports high "agreement" is that they measure LLM-vs-LLM. Measure LLM-vs-human and the optimism collapses. The right metric is the one that hurts.
§ 1 · Why this matters

A benchmark is a claim. Most of the field's claims do not survive their own assumptions.

Three quiet shortcuts dominate the evaluation of code AI today: (i) datasets that overlap with the model's training corpus, (ii) automatic metrics (BLEU, CodeBLEU, exact match) that correlate weakly with developer judgement, and (iii) LLM-as-a-Judge that looks like a human evaluator but agrees with one only 60 % of the time. Each shortcut moves a paper forward. Stacked, they produce a literature in which everything works and nothing replicates.

The lab's third thread is about restoring honest evaluation as a load-bearing element of the field. We re-run baselines under controlled protocols. We pair every automatic metric with a human study. We publish disagreements, not only agreements.

The right metric is the one that you would still publish if it disagreed with you.

We also study causality: does the intervention cause the effect, or do both follow from a confounder? In code AI, the confounder is usually data leakage, prompt sensitivity, or evaluator drift. Counterfactual analyses isolate the signal; we ship them as artefacts.

The lab's most-read paper, the TOSEM 2024 reflection "From Triumph to Uncertainty", argues that the field's perceived progress is partly real and partly a measurement illusion. The papers in this thread try to do something about it.

Three quiet shortcuts the literature relies on
  • Train-test contamination
    Public benchmarks overlap with public code. Headline accuracy on HumanEval includes solutions the model has seen verbatim.
  • Auto-metrics as proxies
    BLEU/CodeBLEU correlate with human ratings around r = 0.4 — better than random, far from sufficient.
  • LLM-as-a-Judge
    When the judge and the judged share a training distribution, the agreement is inflated by 20+ points relative to humans.
Editorial note. Antonio's IEEE Computer column (Jan 2026, vol. 59 no. 1), "Human or Machine? Rebuilding Trust in the Age of AI Text Generation", develops the broader version of this argument for a non-specialist audience.
§ 2 · What is hard

Three reasons reliability is not just a checklist.

2.1 · Drift

Today's judge is not tomorrow's judge.

LLM evaluators drift as their underlying models are retrained. A score reported in 2024 is not directly comparable to a score reported in 2026 with "the same" judge. We propose judge-versioning as a community norm.

2.2 · Counterfactuals are expensive

The right comparison is rarely the cheap one.

"Does this fix fail because of length, or because of the variable name?" — answering needs a controlled rewrite of the input. Tools to generate counterfactuals for code are immature. The lab is building them.

2.3 · Human time

Studies that need humans, need humans.

There is no shortcut around it. Reliable evaluation of developer-facing AI requires developer studies — protocols, IRB, real-time compensation. We design these; we publish the protocols; we welcome co-authors with subject access.

§ 3 · How we ensure it

Method — four pillars of an honest evaluation.

  • Decontamination first
    We hash test sets against public training corpora and discard overlap before reporting.
  • Judge calibration
    Every LLM judge we use is reported with its agreement against a small but real human gold set. If we don't have one, we collect one.
  • Counterfactual perturbations
    For every claimed causal effect, we ship the minimal input edit that should flip the outcome — and report whether it does.
  • Disagreements published
    When our results contradict prior work, we say so and we explain — in dedicated subsections, not footnotes.
honest_eval.py · excerpt 
# Calibrated judging: never report LLM-judge score
# without its human-agreement context.
from aura.eval import HumanGold, LLMJudge, Decontaminator

gold = HumanGold.load("my-task", n=120)
judge = LLMJudge("gpt-4-2024-04")

# 1. Calibrate against humans.
agreement = judge.calibrate(gold)        # Cohen's κ; agreement %

# 2. Decontaminate the test set.
clean = Decontaminator.run(my_test, against="the-stack")

# 3. Report jointly.
score = judge.score(model_outputs(clean))
print(f"score = {score:.2f}   "
      f"(judge κ-human = {agreement.kappa:.2f}; "
      f"test cleaned = {clean.kept_pct:.0%})")

§ 5 · For practitioners

What you can use today.

  • aura-lab/llm-judge
    Reference implementation + harness for judge calibration. Plug your task in, get an agreement-with-humans report on day one.
  • Decontamination toolkit
    Hash your test set against the Stack / GitHub / OSS corpora. Returns the overlap, the source files, and a cleaned subset.
  • Human-study protocol pack
    IRB-ready protocol templates for code-AI user studies. Compensation guidance, sampling, statistical pre-registration.
§ 6 · For researchers

Open problems.

  • Cross-judge calibration
    A unified scale across LLM judges so scores from different evaluators are directly comparable across years.
  • Counterfactual corpora for code
    Public corpora of minimally-edited code pairs labelled by which semantic property changed. Seeded by the lab; contributions welcome.
  • Long-tail metrics
    Most metrics summarise the mean. The interesting failures live in the tail. We need standard, agreed-upon tail metrics for code-AI evaluation.
  • Pre-registration in SE+AI
    Pre-registered hypotheses and analysis plans are standard in psychology and increasingly in CS. The SE+AI community should adopt them; the lab will host a registry.
