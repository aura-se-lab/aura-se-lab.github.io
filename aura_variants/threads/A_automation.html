Thread A · AutomationAI agents for the software lifecycle

Agents that plan, act, and report. Multi-step automation with measurable handoffs.

A single-shot model that proposes a patch is a tool. An agent that retrieves the right files, runs the tests, reads the failures, and revises until it ships is a colleague. The lab studies how to build the second one — and how to evaluate it without lying about success.

§ 1 · Why this matters

The next interface to software is not a chat box. It is a colleague.

Most "AI for code" today lives in an autocomplete: you type, the model proposes, you accept or reject. That product is real, but it is also the easiest version of the problem — the model never has to plan, never has to recover from a failure, never has to coordinate with another agent or another system.

The interesting product, and the harder research, is the agent that shoulders a multi-step task: triage an incoming bug report, reproduce the failure, propose a fix, run the test suite, read the results, revise, and either ship a pull request or escalate. Every link in that chain has to be measured. Every link has to be auditable. Every link has to be cheap enough to be worth running.

An agent without an audit trail is a liability disguised as a tool.

Recent benchmarks like SWE-bench and SWE-Lancer have given the field a target. Our argument is that the field has rushed to score numbers on those benchmarks while skipping over the part where the agent can explain what it did and why. Without that, a 30 % issue-resolution rate is a 30 % automated PR-noise rate.

The lab's automation thread is about building agents whose work is auditable end-to-end, whose handoffs to humans are explicit, and whose failure modes are reported, not absorbed.

The six canonical stations
  1. 01
    Observe
    Read the failing trace, the relevant files, recent commits.
  2. 02
    Plan
    Decompose the task into checkable steps. Estimate the cost.
  3. 03
    Act
    Edit code; call tools; query the test runner. Every action logged.
  4. 04
    Measure
    Inspect the result. Did the test pass? Did anything else break?
  5. 05
    Reflect
    If wrong, why? Write a postmortem the next iteration can use.
  6. 06
    Emit
    Open the PR — or escalate. Always with a human-readable rationale.
§ 2 · What is hard

Three reasons SE agents are not yet real colleagues.

2.1 · Repository state

The world changes under the agent.

Mainline moves while the agent works; tests are flaky; a passing test today fails tomorrow for unrelated reasons. Agents need to model the repository as a moving object, not as a fixed text corpus.

2.2 · Multi-agent coordination

Two agents are not twice one agent.

Hand-offs between a planner, a coder, a tester, and a reviewer introduce coordination overhead and shared-context bugs. We need formal accounts of what each agent commits to.

2.3 · Honest evaluation

Pass@1 hides almost everything.

An agent that passes the unit test by deleting it is "successful" on most leaderboards. Real evaluation needs to inspect the trace — and to penalise shortcuts. This is where our reliability thread (R) meets automation (A).

§ 3 · How we ensure it

Method — auditable agents with measurable handoffs.

  • Every action is a tool call
    Reads, writes, tests, web look-ups, sub-agent invocations — all routed through a typed tool bus that logs everything.
  • Handoffs are commitments
    When a planner hands a task to a coder, it commits in writing to the spec. The coder rejects the task if the spec is malformed. Both events are logged.
  • Trace-aware evaluation
    An agent that "succeeds" by deleting tests is failed by our scorer. The metric inspects the trace, not just the final state.
  • Human-escalation budget
    Agents have a fixed budget of "ask the human" tokens per task. Spending the budget is a feature, not a failure.
agent.py · excerpt 
# Auditable multi-step agent: every action is a logged tool call.
from aura.agents import Agent, Spec, Trace, Tools

a = Agent(
    plan_model="distill-220m",
    code_model="codellama-13b",
    tools=Tools.repo("my-org/my-repo"),
    human_budget=3,           # tokens for escalation
)

trace = Trace.record()
spec = a.plan(issue="ValueError raised on empty input", trace=trace)
patch = a.act(spec, trace=trace)              # commits to spec
result = a.measure(patch, trace=trace)
if not result.ok:
    a.reflect(result, trace=trace)            # write postmortem

# An honest scorer reads the trace, not just the outcome.
from aura.eval import TraceScorer
score = TraceScorer().run(trace)
# Penalises: test deletion, scope creep, shortcuts, unverified rationale.
print(score.report())

§ 5 · For practitioners

What you can use today.

  • aura-lab/agent-bus
    Drop-in typed tool bus for multi-step code agents. Pluggable for any HuggingFace or API model. Trace logs ship with every run.
  • TraceScorer
    Score agent runs against shortcut patterns: test deletion, scope creep, silent dependency injection. Used as a CI gate.
  • Handoff spec language
    A small DSL for describing what one agent commits to deliver to another. YAML or JSON; type-checked.
§ 6 · For researchers

Open problems.

  • Coordination calculus
    A formal theory of inter-agent commitments and conflict resolution. Currently we use ad-hoc protocols; we want a calculus.
  • Long-horizon benchmarks
    SWE-bench tasks finish in 1-3 actions on average. Real engineering tasks take 10-50. We need benchmarks at that horizon, plus methods that scale to it.
  • Curriculum for agents
    How do agents learn from their own past traces? An open question with strong tie-ins to RL from execution feedback.
  • Human-in-the-loop UX
    When should an agent escalate? How should it ask? When the agent says "I'm 70% sure", what does the human actually need to see?
